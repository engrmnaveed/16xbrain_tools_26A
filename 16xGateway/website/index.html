16xGateway is a security sidecar that tokenizes PII before third-party code sees it, runs that code in a hard V8 isolate with no ambient capabilities, and rejects hostile plugins at admission — before a single line executes.

No cloud calls. No AI at runtime. HTTP/1.1 over a Unix domain socket.

The problem

Outsourcing a task shouldn't mean handing over raw PII

When you let a partner's plugin enrich, score, or transform your records, you normally trust their code not to read your secrets, touch your disk, or phone home. 16xGateway removes that trust requirement.

🎭

PII is tokenized in

Emails, phones, cards, SSNs and secret-keyed fields become deterministic [TOKEN_MASK_SHA256_…] tokens before the plugin runs — and real values are restored on the way out.

📦

No capabilities by default

The isolate is ECMAScript intrinsics only. No require, no process, no fs, no network, no timers. Egress is a host-side, allowlisted ctx.fetch.

🛑

Hostile code never runs

An AST gate refuses admission to code that references eval, process, off-list require, dynamic import and more — with precise reason codes and zero execution.

Architecture

The request pipeline

Step order is normative: sanitization always precedes execution, and the per-request reverse map is destroyed on every path — success, timeout, error, rejection.

Host SDK

Three lines to adopt

Branch on one status field. Operational failures never throw.

import { Gateway } from "@16xbrains/gateway-client";

const gateway = new Gateway({ socket: "/var/run/16xgateway.sock" });

const result = await gateway.execute("outsourced-analytics", record);
if (result.status === "success") save(result.data); // real values restored

Plugin SDK

Agencies write one file

One CommonJS file, one SDK import, tested locally against the real gate and isolate.

const { definePlugin } = require("@16xbrains/plugin-sdk");

module.exports = definePlugin({
  id: "outsourced-analytics", version: "1.2.0",
  handler: async (payload, ctx) => {

    return { ...payload, score: scoreOf(payload) };

Honest by design

What it defends against — and what it doesn't

Defends against

Plugin code reaching for process.env, fs, network or the module graph (absent from the isolate); raw PII exposure (tokenized, per-request reverse map); eval-family & prototype-pollution injection (AST gate + isolate); runaway loops and allocation bombs (timeout + memory ceiling); tampered files at rest (hash re-verification); network exfiltration (no egress primitive; allowlisted, size-capped fetch).

Does NOT defend against

A V8 isolate escape (0-day); PII regex false negatives — this is not a DLP system; determined obfuscation vs the gate (the isolate is the real boundary); exfiltration to domains you explicitly allowlist; a compromised gateway host; and isolated-vm maintenance-mode risk.

16xGateway is heuristic tokenization to keep casual and hostile plugin code away from obvious identifiers. It does not guarantee complete PII coverage and must not be represented as data-loss prevention.
