Real test data, without the risk

Seeding a dev database usually means a stale prod dump, a flaky faker script, or hours hand-writing INSERTs. ForgeDB plans your FK graph like a real migration tool and generates integrity-guaranteed data at millions of rows per second — fully offline.

🧭

Topological planner

Kahn's algorithm orders every table so parents always exist before children. Self-references (manager_id) and multi-table cycles are detected and resolved automatically — a deferred edge gets a second-pass UPDATE patch.

🔗

Integrity by construction

Foreign keys are only ever drawn from a parent's completed key pool — broken references are unrepresentable, not just unlikely. Every run finishes with a real pragma_foreign_key_check audit.

🎲

Deterministic engine

Seeded PRNG streams derived per table and per column. Same schema + seed = byte-identical dataset on any machine. Share a 20-char seed with your team instead of a 2GB dump.

📈

Realistic distributions

Uniform, zipf power-law hotspots ("few customers place most orders"), round-robin (every parent gets children), or one-to-one bijective mapping — configurable per FK edge.

⚡

Built for throughput

Compiled per-column closures, 10k-row streaming batches, WAL mode, multi-row prepared INSERTs, and a Rust bridge for the desktop app — ~2.7M rows/sec in-engine.

🔒

Private by design

Every byte comes from a bundled procedural engine. No LLM, no cloud API, no network call — your schema and data never leave your machine.

FAQ

Is it really free?

Yes. It's a showcase of how we think about data infrastructure at 16xBrains. No account, no key, no usage limits.

Does it call any AI or cloud service?

No. ForgeDB has zero runtime LLM/cloud/network dependency. Every byte of generated data comes from a deterministic procedural engine bundled in the app.

What column types are supported?

increment, uuid, fk, names, emails, phones, addresses, companies, words/sentences, numbers, dates, enum, pattern (SKU-####-AA), and {slot} templates.

How does it handle circular foreign keys?

Self-references resolve in-table by sampling only earlier rows. Multi-table cycles break at a nullable/deferrable edge: pass one writes NULL, pass two patches in the real value — all set-based, no per-row updates.

Need a real data platform?

ForgeDB seeds a local SQLite file for dev and testing. Designing and operating production data infrastructure — pipelines, warehouses, migrations — is what we do.
